\chapter{Introduction}
\label{ch:intro}

\section{Contexte du projet}

\nutritrack{} est une plateforme d'ing\'enierie de donn\'ees de bout en bout
d\'edi\'ee au suivi nutritionnel. Le projet collecte des donn\'ees de produits
alimentaires depuis \off{}, les nettoie et les transforme via des pipelines
ETL automatis\'es, les stocke dans un entrep\^ot de donn\'ees en sch\'ema
en \'etoile, et les expose \`a travers une API REST consomm\'ee par une
interface Streamlit. Un data lake bas\'e sur MinIO impl\'emente l'architecture
m\'edaillon (bronze/silver/gold) avec une gouvernance compl\`ete et une
conformit\'e \rgpd{}.

\section{Objectifs}

Le projet a pour objectif de d\'emontrer la ma\^itrise des 21 comp\'etences
(C1--C21) requises pour la certification \textit{Expert en Infrastructures
de Donn\'ees Massives} (RNCP37638). Il couvre les 4 blocs de comp\'etences
\`a travers un projet unique et coh\'erent.

\textbf{Objectifs SMART :}
\begin{itemize}[nosep]
  \item \textbf{S} : Construire une plateforme de suivi nutritionnel
        int\'egrant 5 types de sources de donn\'ees
  \item \textbf{M} : Traiter 100\,000+ produits avec 95\,\%+ de
        compl\'etude sur les champs nutritionnels cl\'es
  \item \textbf{A} : Utiliser des technologies open-source dans un
        environnement Docker Compose
  \item \textbf{R} : S'aligner sur le mod\`ele de donn\'ees \off{} et
        les normes nutritionnelles europ\'eennes
  \item \textbf{T} : Livrer en un cycle de 3 mois (12 sprints d'une semaine)
\end{itemize}

\subsection{Contexte et besoin m\'etier}

\nutritrack{} r\'epond \`a une demande croissante de suivi nutritionnel
personnalis\'e. Le besoin m\'etier provient de trois groupes de parties
prenantes : (1)~les utilisateurs finaux souhaitant suivre leur alimentation
et recevoir des alternatives plus saines, (2)~les nutritionnistes ayant
besoin de donn\'ees alimentaires agr\'eg\'ees pour la recherche et les
recommandations, et (3)~les administrateurs g\'erant la plateforme et
assurant la qualit\'e des donn\'ees.

L'\'etude de faisabilit\'e a identifi\'e \off{} comme source de donn\'ees
principale --- une base collaborative et open-source de produits alimentaires
avec plus de 3~millions d'entr\'ees, incluant les notations Nutri-Score et
des valeurs nutritionnelles d\'etaill\'ees.

\subsection{Note de synth\`ese}

\textbf{Analyse du besoin :} Le projet r\'epond au besoin d'une plateforme
centralis\'ee combinant des donn\'ees de produits alimentaires provenant de
sources multiples (API, exports bulk, web scraping) avec un suivi de repas
personnel.

\textbf{Actions de conformit\'e \rgpd{} :} Inscription bas\'ee sur le
consentement, limites de r\'etention des donn\'ees, pseudonymisation SHA256
dans l'entrep\^ot, nettoyage automatis\'e via
\texttt{rgpd\_cleanup\_expired\_data()}.

\subsection{\'Etude technique d'architecture (\competence{C3})}

\subsubsection{Analyse fonctionnelle}

Le syst\`eme doit : (1)~collecter des donn\'ees de produits alimentaires
depuis 5 sources h\'et\'erog\`enes, (2)~nettoyer, normaliser et d\'edupliquer
les donn\'ees, (3)~stocker les donn\'ees structur\'ees dans PostgreSQL,
(4)~exposer les donn\'ees via une API REST s\'ecuris\'ee, (5)~fournir une
interface de suivi nutritionnel, et (6)~maintenir un data lake avec
architecture m\'edaillon.

SELECT p.barcode, p.product_name, b.brand_name,
       p.nutriscore_grade,
       ts_rank(to_tsvector('french', p.product_name),
               plainto_tsquery('french', 'chocolat noir'))
               AS relevance
FROM app.products p
LEFT JOIN app.brands b ON p.brand_id = b.brand_id
WHERE to_tsvector('french', p.product_name)

  \item \textbf{P4:} Added to sprint backlog, prioritized during sprint planning.
\end{itemize}

\textbf{Escalation:} L1~(Engineer) $\to$ L2~(Senior Engineer, if unresolved within response time) $\to$ L3~(Project Lead, P1$>$2h or P2$>$4h).

\section{Le\c{c}ons apprises}

\begin{enumerate}[nosep]
  \item \textbf{PostgreSQL unique OLTP+OLAP} fonctionne \`a petite \'echelle
        gr\^ace \`a l'isolation de sch\'emas, mais n\'ecessiterait une
        s\'eparation \`a plus grande \'echelle.
  \item \textbf{DuckDB} est remarquablement efficace pour l'analytique
        big data sans infrastructure d\'edi\'ee.
  \item \textbf{CeleryExecutor d'Airflow} ajoute de la complexit\'e mais
        permet le parall\'elisme ETL.
  \item \textbf{SCD Type~2} requiert une logique \texttt{IS DISTINCT FROM}
        soigneuse pour les valeurs NULL.
  \item La \textbf{conformit\'e \rgpd{}} doit \^etre con\c{c}ue d\`es le
        d\'epart, pas ajout\'ee a posteriori.
\end{enumerate}


\section{Am\'eliorations futures}

\begin{itemize}[nosep]
  \item S\'eparation OLTP/OLAP avec un moteur analytique d\'edi\'e
        (ClickHouse, Redshift)
  \item Migration vers un catalogue de donn\'ees complet (DataHub)
  \item Impl\'ementation de tests de qualit\'e avec Great Expectations
  \item Pipeline CI/CD pour les DAGs Airflow
  \item Monitoring avanc\'e avec Grafana et alerting PagerDuty
  \item Streaming en temps r\'eel avec Apache Kafka
\end{itemize}
